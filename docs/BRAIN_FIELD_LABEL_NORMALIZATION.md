# Brain field label normalization

## Problem

Audits showed `brain_fields.unique_normalized_labels ≈ 1` while `label` and `raw_json` still hold useful worksheet text. Executive/API routes filter on `normalized_label` (e.g. `%sq ft%`, `%color%`), so blank or uniform normalized labels hide real field data.

## Where rows are written

1. **Sync** (`src/morawareDiscovery.js`) builds job objects with `forms[].fields[]` from `normalizeJobFormsFromSdkParsed` (`src/morawareJobFormsSdk.js`). Each in-memory field includes `label` and `normalizedLabel` (# Moraware “friendly” normalization: lowercase, `sq ft` spelling, etc.).
2. **Supabase insert** happens in `upsertBrainFields` (`backend-core/src/brain/supabaseBrainStore.js`), which maps each field row into table columns.

### Root cause (historical behavior)

Previously, `brain_fields.normalized_label` was set **only** from `String(fld?.normalizedLabel ?? "")`. If the in-memory object lost `normalizedLabel` (wrong key, partial object, or older pipeline), Postgres stored an **empty** string while `label` and `raw_json` were still populated. The Moraware XML/query layer was never at fault—this was entirely how we mapped the JS object into Supabase.

`normalizeLabel` / `normalizeMorawareLabel` in the ingest path were never too aggressive for storage; the issue was **relying on a single optional property** instead of deriving from `label` + embedded Moraware nodes in `raw_json`.

## Canonical storage format

`brain_fields.normalized_label` is now a **stable snake_case search key**, generated by `computeNormalizedLabelForBrainFieldRow` (`backend-core/src/brain/brainFieldNormalizedLabel.js`):

- **Sources (in order):** human `label` → `name` / `fieldName` → XML-ish paths inside `raw_json` → Moraware `normalizedLabel` / `normalized_label` as a fallback → `field_<id>` if nothing else exists.
- **Rules:** lowercase; unify `Sq.Ft.` / `sq ft` / `sqft` → token `sqft`; `square feet` → `square_footage`; non-alphanumerics → underscores; collapse repeats.

This does **not** modify `value`, `numeric_value`, or `raw_json`, and does **not** change worksheet Sq.Ft. aggregation in `computeWorksheetSqFtMetrics` (still driven by the original in-memory `label` / `normalizedLabel` on the SDK objects during sync).

## API compatibility

Executive routes that query Sq.Ft. and field trends used `ilike` patterns such as `%sq ft%`. Snake_case keys like `worksheet_sqft` no longer match `%sq ft%`. `backend-core/src/server.js` now ORs additional patterns (`%worksheet_sqft%`, `%sq_ft%`, `%square_footage%`, etc.) so existing dashboards keep working after backfill.

## Backfill

`backend-core/src/scripts/backfillBrainFieldNormalizedLabels.js` recomputes `normalized_label` **only** where it is null/empty, using the same helper as sync. Default is **dry run** (`DRY_RUN` defaults to `1`); set `DRY_RUN=0` to apply (see `npm run eos:backfill:field-labels`).
